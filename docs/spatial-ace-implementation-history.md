## Mathematical Contract

The strict-equivalent backend must reproduce the existing SUS2-SH SO(3)
coupling graph.  For every current product node

```text
(l1, m1) x (l2, m2) -> (L, M)
```

the numerical map has to equal the same real-basis CG tensor product currently
generated by `RealCGCoeff` in `dev_src/sh_model_init.cpp`.

This is stronger than ordinary scalar spherical-function multiplication.  The
Gaunt projection

```text
int conj(Y_LM) Y_l1m1 Y_l2m2 dOmega
```

has a parity selection rule and is not identical to the current CG tensor
product for all allowed intermediate angular momenta.  A strict spatial ACE
implementation therefore has to represent the same SO(3) tensor product map in
a spatial/SO2-friendly layout; it must not silently replace the model with only
ordinary scalar pointwise products.

The direct spatial ACE variant is allowed to use one-shot spatial invariants,
but it is a new basis unless an explicit proof or test shows equality for a
given low-order case.

## 5.5 Subagent Review Notes

- Mathematical review confirmed that 2302.03655's SO(2) reduction is an exact
  reparameterization only after rotating into an axis-aligned frame where the
  filter-side spherical harmonic has `m=0`.  It is not a generic replacement of
  arbitrary aggregated ACE tensor products by scalar pointwise multiplication.
- Mathematical review confirmed that 2509.14961's spherical spatial ACE is a
  direct spatial/Gaunt product basis.  It should be kept as a separate
  non-equivalent direct variant unless a specific low-order equality is proven.

  first, placed beside the existing SH runtime switches in `mtpr_sh.cpp`, to
  avoid model-format churn while benchmarking exact equivalence.
- Code review recommended centralizing the duplicated `sh_products_` loops
  behind forward/reverse helpers parallel to the existing product-row helpers,
  then extending those helpers to gate's pruned product subset.
